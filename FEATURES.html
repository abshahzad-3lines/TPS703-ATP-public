About this guide

What this is

A single-page test catalog for every feature delivered in Phase 10 (ATP authoring) and Phase 11 (S-parameter workflows) of the TPS-703 ATP Automation System — the web app that runs Acceptance Test Procedures for Northrop Grumman's S-band radar transmitter subsystems (CAGE Code 97942).

Use it as a checklist when reviewing a deployment: each card pairs a feature with the exact steps to verify it works end-to-end. The feature list mirrors the original Phase 10 / Phase 11 work items, so you can tick each one off without cross-referencing tickets or code.

What the app does (one paragraph)

Replaces paper-based ATP workflows with a digital system that enforces test sequencing, captures live instrument measurements, validates them against specification limits, and produces tamper-evident certificates. Phase 10 added a versioned ATP authoring layer with peer review and AI-assisted drafting; Phase 11 added S-parameter (.sNp) import, visualisation, fixture de-embedding, golden-unit compare, and AI sweep analysis.

How to use this guide

  1. Open the production URL above and log in with admin / admin123. (If it takes ~30 s on first hit, that's the Render free-tier waking up — see the cold-start tip at the bottom.)
  2. Pick a feature card from any section. Follow the How to test steps top-to-bottom.
  3. UI features list exact click paths (sidebar entries, tab names, button labels). API features include curl snippets you can paste straight into a terminal.
  4. AI features call Groq (llama-3.3-70b-versatile). They may take 1–4 s per call. If you see 503 the GROQ_API_KEY isn't set; if you see 429 the per-user rate limit (20/min, 200/day) kicked in.

Need an API token for the curl examples?

Run this in your terminal once — it stores the bearer token in $TOK for the rest of your shell session:

URL=https://tps703-atp-cu9j.onrender.com
TOK=$(curl -s -X POST $URL/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"admin123"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")
echo "Token captured, length $(echo -n $TOK | wc -c | tr -d ' ')"

Every curl in this guide assumes $URL and $TOK are already set. The token is valid for 15 minutes — re-run if you get a 401.

Phase 10 — ATP Authoring

State-machine-governed authoring of Acceptance Test Procedures. Sidebar: ATP Author (engineer+).

UI Authoring UI /atp-author

How to test
  1. Sidebar → ATP Author.
  2. Click + New draft, pick a subsystem, fill code + name, submit.
  3. On the detail page, click Add step in the Steps tab — fill name, type, limits.
  4. Five tabs should show: Steps · Metadata · Simulation · History · AI.

SM State machine

How to test
  1. Open a draft ATP. Top-right shows Submit for review.
  2. Click it → state badge flips draft → in_review.
  3. As the author you'll see Withdraw (Approve/Reject hidden).
  4. History tab shows every transition with timestamp + user.

Auth Peer review (two-user gate)

How to test
  1. Submit a draft as admin → state becomes in_review.
  2. Open the browser console: 
    await fetch('/api/auth/login', {method:'POST',headers:{'Content-Type':'application/json'},body:JSON.stringify({username:'peer',password:'pw1234'})}).then(r=>r.json()).then(j=>localStorage.setItem('token',j.access_token));
location.reload();
    (If peer doesn't exist yet, ask an engineer to create one.)
  3. Now action bar shows Approve / Reject.
  4. As admin again, try to Approve — it should return 403 — Author cannot approve their own ATP.

Validator Step-schema validation

How to test
  1. In a draft, add an output_power step with no limits and no frequency.
  2. Top of page shows a red banner listing each issue (missing limit, missing frequency_mhz, missing equipment for role power_meter, …).
  3. Try Publish — blocked until issues clear.

Sim Golden-unit simulation

How to test
  1. Open any published or draft ATP. Switch to the Simulation tab.
  2. Click Run simulation.
  3. Shows pass / fail / skipped counts + a per-step measured-vs-limits table; history table appends a row each run.

UI DOCX / PDF import

How to test
  1. Sidebar → ATP Author → Import .docx / .pdf.
  2. Upload a Word/PDF doc that contains lines like Step 1. …, Step 2. ….
  3. Dialog shows extracted text preview + heuristic step split.
  4. Pick subsystem, confirm code/name, choose heuristic split or AI extract, click Create.

UI Revision diff

How to test
  1. From a published ATP, click Clone (copy icon) in the list view.
  2. Edit one step in the new revision.
  3. From the new revision's detail page, click Diff vs parent.
  4. Page at /atp-author/diff/<base>/<target> shows metadata changes + added/removed/modified step tables.

API Export / import signed bundle

How to test
  1. Detail page → top-right Export bundle → downloads atp-<code>-rev<X>.json.
  2. Open the file — bottom has signature.hmac.
  3. Re-import via API: 
    curl -X POST $URL/api/atp/import \
  -H "Authorization: Bearer $TOK" \
  -H "Content-Type: application/json" \
  --data @atp-MYCODE-revA.json
  4. Edit the bundle (remove a step) and re-POST — server returns 400 signature mismatch.

Phase 10 — AI assists (Groq)

All four are rate-limited (20/min, 200/day per user). Without a GROQ_API_KEY they return 503 with a clear message.

AI Extract steps from document

How to test
  1. ATP Author → Import .docx / .pdf, upload a procedure doc.
  2. In the dialog, pick the AI-extract via Groq radio option.
  3. Click Create — Groq returns structured steps with step_type, frequency, power, limits, units filled in.
  4. The resulting draft's Source badge says AI-extracted.

AI Safety-warning generator

How to test
  1. On a draft ATP, open the Steps tab and click Edit on any step.
  2. In the editor dialog, click the ✨ sparkles button next to the Safety warning field.
  3. Groq drafts a 1-2 sentence warning based on step parameters (RF power, voltage, lifting, ESD, etc).
  4. Edit if needed, click Save.

AI Step-ordering review

How to test
  1. On any ATP, switch to the AI tab.
  2. Click Review in the AI step-ordering review card.
  3. Groq returns concerns categorised as missing_warmup, missing_settling, dependency_violation, duplicate, safety_gap, limit_mismatch, redundant_stimulus.

AI Revision-impact summary (ECR)

How to test
  1. Go to a revision-diff page (see Revision diff above).
  2. Top card: AI impact summary → click Generate summary.
  3. Groq returns 2-4 ECR-ready paragraphs explaining what changed and why it matters, citing specific frequencies / limits.

Phase 11 — S-Parameter Workflows

Touchstone I/O, visualisation, de-embedding, golden-unit compare, masks. Sidebar: S-Parameters (technician+).

IO Touchstone v1/v2 read + write

How to test
  1. Sidebar → S-Parameters → Upload .sNp.
  2. Pick any .s1p/.s2p/.s3p/.s4p file.
  3. New row appears in the list with port count, sample count, frequency span, Z₀.
  4. Click the small download arrow → server returns a normalised Touchstone v2 string.

UI Visualisation — 5 plot types

How to test
  1. Open any uploaded sweep → Plots tab (default).
  2. Use the chart-type dropdown to switch between Magnitude (dB), Phase (°), Group delay (s), Smith chart, Polar.
  3. Toggle the S11 / S12 / S21 / S22 chips to add/remove traces.
  4. Smith + polar are custom Canvas; line plots use Recharts.

Cal OSLT fixture de-embedding

How to test
  1. Upload four cal sweeps (open, short, load, thru) and your raw DUT sweep.
  2. Create a cal set via API: 
    curl -X POST $URL/api/sparam/cal-sets \
  -H "Authorization: Bearer $TOK" \
  -H "Content-Type: application/json" \
  -d '{"name":"CalA","cal_type":"OSLT","open_sweep_id":2,"short_sweep_id":3,"load_sweep_id":4,"thru_sweep_id":5}'
  3. Open the raw DUT sweep → De-Embed tab → pick CalA → Apply calibration.
  4. New de-embedded sweep is created and linked back; source badge says de embedded.

UI Golden-unit compare + overlay

How to test
  1. Upload your reference sweep. Register it via API: 
    curl -X POST $URL/api/sparam/golden-refs \
  -H "Authorization: Bearer $TOK" \
  -H "Content-Type: application/json" \
  -d '{"name":"K245 golden","subsystem_id":1,"sweep_id":1,"uut_family":"K245"}'
  2. Upload a second (measured) sweep. Open it → Compare tab.
  3. Pick the golden ref from the dropdown → Compare.
  4. Top chart overlays measured + golden; bottom chart plots Δ|S| (dB) per frequency.

VNA VNA auto-archive

How to test
  1. (Lab-only) Run a test step that exercises a network-analyzer step type.
  2. The execution runner calls sparam_capture.archive_capture() with the captured matrix.
  3. Open S-Parameters; new row with source = captured and test_run_id populated.
  4. To unit-test without hardware: POST a sweep with an explicit test_run_id param to the upload endpoint — auto-archived sweeps look identical to that.

API Export to .mat / .npz / .csv

How to test
  1. Open any sweep → top-right buttons: .s2p · .csv · .mat · .npz.
  2. Each downloads instantly.
  3. .csv opens in Excel/Numbers (one row per frequency, real/imag columns).
  4. .mat opens in MATLAB: load('sweep.mat'); plot(f, 20*log10(abs(squeeze(s(:,2,1)))))
  5. .npz opens in Python: d = np.load('sweep.npz'); plt.plot(d['f'], 20*np.log10(np.abs(d['s'][:,1,0])))

IO Touchstone validation (line-precise errors)

How to test
  1. Create a deliberately-broken file (e.g. wrong column count): 
    # HZ S MA R 50
2.8e9 0.1 0
2.9e9 0.1 0
    (only 3 numeric columns for a 2-port file → should be 9)
  2. Upload it through the UI.
  3. A red alert appears: line 2: expected 9 columns for 2-port (MA format), got 3.
  4. Other test cases: non-numeric token (line X, col Y), non-monotonic frequency, missing # option line.

UI Pass/fail mask + chart overlay

How to test
  1. Open a sweep → Mask tab → + New mask.
  2. Fill f_start = 2.85, f_stop = 3.05, param = s21, quantity = mag_db, min = -3, max = 1, name it, Create.
  3. Pick the mask from the dropdown → Evaluate.
  4. Alert shows PASS / FAIL; chart overlays the pass window as an amber band; table lists each band's worst value + frequency.

Phase 11 — AI assists (Groq)

Four cards in the sweep's AI tab. Same 20/min, 200/day per-user rate limit as Phase 10 AI.

AI Anomaly detection vs history

How to test
  1. Upload at least 3–5 sweeps for the same subsystem (so history exists).
  2. Upload one obviously-broken sweep (e.g. ~6 dB suckout at 2.95 GHz).
  3. Open the broken sweep → AI tab → Detect in the anomaly card.
  4. Returns severity-tagged entries: resonance, suckout, ripple, slope_deviation, phase_glitch, in-band-loss, match_degrade.

AI Narrative vs golden

How to test
  1. Register a golden ref (see Compare feature).
  2. On a different sweep, AI tab → Narrative vs golden card → pick the golden → Generate.
  3. Groq returns 2-3 paragraphs explaining the diff in plain English with frequency / dB citations.

AI Cal-set suggestion

How to test
  1. Register a few cal sets (different frequency spans).
  2. Upload a raw sweep. AI tab → Cal-set suggestion card → Suggest.
  3. Returns best_match_id, confidence, and a one-sentence reason — typically based on frequency-span overlap and Z₀ match.

AI Mask-failure explanation

How to test
  1. Build a mask that the current sweep will fail.
  2. AI tab → Mask-failure explanation → pick the mask → Explain.
  3. Groq returns one paragraph naming the band, the actual-vs-limit magnitude, and a plausible lab-engineer root cause.

Non-feature work that's still worth knowing about.

Limit Per-user AI rate limiter

How to test
  1. Open the browser console while logged in.
  2. Fire 22 AI calls in parallel: 
    const tok = localStorage.getItem('token');
const calls = Array.from({length: 22}, (_, i) =>
  fetch('/api/atp/ai/extract-from-document', {
    method: 'POST',
    headers: {'Content-Type':'application/json', Authorization: `Bearer ${tok}`},
    body: JSON.stringify({text:'Step 1. test', subsystem_id:1, code:`B-${i}`, name:`B${i}`}),
  }).then(r => r.status));
console.log(await Promise.all(calls));
  3. You'll see ~20 successes + 2 × 429 from the local limiter. Some of the successes may further return 502 from Groq's own per-second cap.
  4. Limits are 20 / minute and 200 / day per user (see services/rate_limit.py).

Auto Env auto-load from ~/Desktop/.env.local

How to test
  1. Backend config.py loads (in order, first-found-wins): backend/.env → repo/.env → ~/Desktop/.env.local.
  2. Run the backend with no shell env exported — AI endpoints still work because GROQ_API_KEY auto-loads from the global file.
  3. Remove the key from all three locations, restart — AI endpoints return 503 with a clear message; the rest of the app still works.

If the first request is slow or fails — cold-start tip

Render's free tier suspends the container after 15 minutes of inactivity. The first request after that wakes it up in ~30 s. If a login fails with a network error, refresh once more — it'll come back. To eliminate the sleep, upgrade the Render service to the $7/mo Starter plan.

Hard cap protection: the service is locked to the free plan (plan=free, 1 instance, no preview envs). Even with a card on file you can't be charged for compute unless someone explicitly upgrades the plan in the dashboard. Bandwidth (100 GB/mo) and build minutes (500/mo) auto-suspend rather than bill.
